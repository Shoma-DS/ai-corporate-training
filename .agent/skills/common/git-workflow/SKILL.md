# Git ワークフロースキル (AI法人研修リポジトリ版)

## 絶対パスルール（必須）
- ユーザーに Git コマンドを見せるときは、固定のパスではなく `WORKSPACE_ROOT` を使って絶対パスを組み立てる。
- `WORKSPACE_ROOT` は `/Users/deguchishouma/Desktop/AI法人研修` です。
- `git status` のように短く書かず、`git -C "$WORKSPACE_ROOT" status` の形で渡す。
- `cd` を使う場合も、移動先は `"$WORKSPACE_ROOT/..."` の形にする。

## 事前確認 (Pre-flight Check)
- スキル実行前に必ず `git -C "$WORKSPACE_ROOT" status` を確認し、未コミットの変更があるか把握すること。
- ステージング（`git add`）の前に、`非公開/`、出典PDF、`.DS_Store`、個人情報や社内固有情報、および機密データが含まれていないかを厳重に確認する。特に `非公開/` は Git 管理外であるため、絶対にコミットに含めてはいけません。

## コミットメッセージのルール（厳守）
コミットメッセージは日本語で作成し、以下のフォーマットにすること。専門用語を避け、小学生でも理解できる簡単な表現を使う。

```text
<1行要約: 小学生でもわかる短い言葉で書く>

<詳細: 小学生でもわかる言葉で、何をしたかを細かく書く>
- どこを変えたか
- 何をしたか
- 何がよくなるか
（詳細部分は3〜5行で書くこと。短すぎる説明は避ける）
```

### カタカナ語や専門用語の言い換えルール
- `リファクタリング` → `書き方を整理した`
- `依存解決` → `足りない道具を入れた`
- `最適化` → `むだな動きを減らした`
- `クロスプラットフォーム対応` → `いろいろなパソコンで動きやすくした`

## コミット前の説明と承認 (`git commit`)
- 未コミットの変更がない場合はこのステップをスキップする。
- コミットを実行する前に、以下の手順を踏むこと。
  1. `git -C "$WORKSPACE_ROOT" diff --staged` 等で変更内容を確認する。
  2. ユーザーに対して、今回の変更内容を小学生にもわかるように説明する。
  3. 作成したコミットメッセージ案（要約 + 詳細）を提示する。
  4. 「この内容でコミットして良いですか？」と承認を求める。
- 承認が得られた場合のみ、コミットを実行する。

### コミット前説明の型（必須）
```text
<1行での要約>

全体像:
<この変更が何のためか>
<何がそろうのか、何が良くなるのか>

変更したファイル:
- <ファイル名>: <どのように変えたか>. <どう役立つか>
- <ファイル名>: <どのように変えたか>. <どう役立つか>
```

## リモートとの同期と反映 (`fetch` / `pull --rebase` / `push` / PR)
手元の変更をリモートへ反映するためのフローです。本リポジトリでは原則として `main` に直接コミットして push します。ただし、作業ブランチや PR は、ユーザーが明示した場合、または破壊的・大規模な変更で確認導線が必要な場合に限ります。

**【手順A】まず `fetch` で更新の有無を見る:**
- 最初に `git -C "$WORKSPACE_ROOT" fetch origin` を実行する。
- リモートに更新がない場合は、`pull --rebase` は省略してよい。
- 更新がある場合だけ、次のスタッシュと `pull --rebase` の手順へ進む。

**【手順B】pull が必要なときだけ未コミット変更を避難する:**
- `git -C "$WORKSPACE_ROOT" status --porcelain` で未コミットの変更があるか確認する。
- 変更がある場合は、一旦スタッシュに避難させる。
```bash
git -C "$WORKSPACE_ROOT" stash push -u -m "automated git-workflow stash"
```

**【手順C】更新があるときだけ `pull --rebase` する:**
- リモートの更新がある場合だけ `main` または現在のアクティブブランチに取り込む。
```bash
git -C "$WORKSPACE_ROOT" pull --rebase origin $(git -C "$WORKSPACE_ROOT" branch --show-current)
```
- 競合が発生した場合:
  1. `git diff` 等で競合箇所を確認する。
  2. ユーザーに対し、「ローカルの変更」と「リモートの変更」の内容を簡潔かつ具体的に説明する。
  3. 「どちらを優先しますか？」とユーザーに聞き、指示に従って解決する。
  4. 解決後、`git add` して `git rebase --continue` を行う。

**【手順D】反映（Push / PR）:**
- rebase が完了した場合、または `fetch` の時点で更新がなかった場合は push を行う。
- 原則として、現在のアクティブブランチ（通常は `main`）に push する。
```bash
git -C "$WORKSPACE_ROOT" push origin $(git -C "$WORKSPACE_ROOT" branch --show-current)
```
- ※ただし、ユーザーから明示的に指示があった場合や、大規模変更で確認が必要な場合は、作業ブランチを push し、GitHub CLI (`gh`) を使ってプルリクエストを作成します。
```bash
gh pr create --base main --head $(git -C "$WORKSPACE_ROOT" branch --show-current) --title "<PR タイトル>" --body "<PR 説明>"
```

**【手順E】避難した変更を戻す:**
- 手順Bでスタッシュした場合は、ここで戻す。
```bash
git -C "$WORKSPACE_ROOT" stash pop
```
- `stash pop` で競合が起きた場合も、差分の違いを説明してからユーザー確認を取る。
