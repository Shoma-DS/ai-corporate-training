# Google Workspace・GASで進めるAI業務効率化/DX実践講座 スライド構成案

## Deck Concept

中小企業がすでに使っているGoogle Workspaceを土台に、GASと生成AIを組み合わせて、業務課題の整理から小規模な自動化プロトタイプ、運用設計、導入提案まで進める講座。

AX・DXワークショップ講座のカリキュラム画像から読み取れる「基礎理解 -> 技術選定 -> 評価 -> 提案書作成」の流れを、本講座では「業務棚卸し -> Workspace設計 -> GAS自動化 -> AI活用 -> 運用設計 -> 提案書作成」に置き換える。

## 第4回: 生成AIを組み込んだ文書作成・分類・要約

画像カリキュラムの「データサイエンス基礎」は、生成AI、LLM、RAG、統計解析、機械学習の違いを理解し、分析や予測の使い分けを学ぶ回として設計されている。本講座ではこの考え方を、AIが得意な文書処理と人が確認すべき判断工程に置き換える。

| No. | スライドタイトル | 主な内容 |
| ---: | --- | --- |
| 4-1 | AI・データサイエンス・GASの役割の違い | GASは処理、AIは生成・分類・要約、人は確認 |
| 4-2 | 生成AIが得意なこと・苦手なこと | 文章作成、要約、分類、表現調整、根拠確認の限界 |
| 4-3 | 業務で使うプロンプト設計 | 役割、前提、入力、出力形式、禁止事項、判断基準 |
| 4-4 | 問い合わせ分類と返信案作成 | カテゴリ分類、優先度、返信ドラフト、人の承認 |
| 4-5 | 議事録・日報・報告書の要約 | 情報整理、抜け漏れ確認、次アクション抽出 |
| 4-6 | AI出力の確認手順 | ハルシネーション、機密情報、著作権、根拠確認 |
| 4-7 | 演習: 確認付きAI業務フローを作る | プロンプトと確認基準をセットで作成 |

## 第5回: AI自動化の要件定義・運用設計

画像カリキュラムの「データサイエンス応用」は、レコメンド、検索、画像解析、数理最適化など、課題に応じた技術選定と評価方法を扱う回として設計されている。本講座では、業務分類、通知条件、優先順位、リマインド、AI分類結果の評価に置き換える。

| No. | スライドタイトル | 主な内容 |
| ---: | --- | --- |
| 5-1 | AI自動化は「作って終わり」ではない | 継続運用、保守、例外対応まで設計する |
| 5-2 | 要件定義で決めること | 対象業務、利用者、入力元、出力先、処理条件 |
| 5-3 | 業務分類・優先順位・通知条件 | レコメンドや検索の考え方を業務判断に応用 |
| 5-4 | ログと評価指標 | 正しく分類できたか、通知漏れはないか、削減時間は出たか |
| 5-5 | 情報管理と権限 | 個人情報、顧客情報、機密情報、共有範囲 |
| 5-6 | GASの制限とリスク | 割り当て制限、大量処理、停止時対応、手動復旧 |
| 5-7 | 演習: 運用設計書とリスクチェックリストを作る | 本番運用前の確認項目を整理 |

## 第6回: AI業務効率化プロジェクト提案書の作成

画像カリキュラムの「DX・AX提案書の設計」は、ユースケース、業務フロー、KPI、課題優先順位、To-Be、技術アプローチ、ロードマップを提案書にまとめる回として設計されている。本講座でもこの構造を採用し、Google Workspace/GASの導入提案書として完成させる。

| No. | スライドタイトル | 主な内容 |
| ---: | --- | --- |
| 6-1 | 提案書は「作ったものの説明」ではなく「導入判断の材料」 | 背景、課題、効果、リスク、次アクション |
| 6-2 | ユースケースを明確にする | 誰の、どの業務を、何のために改善するか |
| 6-3 | 業務フローとTo-Be像 | As-Is、課題、To-Be、Gap |
| 6-4 | 技術アプローチ | Sheets、Forms、Gmail、Drive、Calendar、GAS、生成AI |
| 6-5 | KPIと効果試算 | 削減時間、対応速度、ミス削減、品質安定、費用対効果 |
| 6-6 | ロードマップ | 試行、本番化、教育、運用、改善 |
| 6-7 | 発表・レビュー | ミニプロトタイプ、運用設計、リスク、導入効果を説明 |
