# Google Workspace・GASで進めるAI業務効率化/DX実践講座 講師用メモ

## 講座全体の話し方

この講座は、Google WorkspaceやGASの機能紹介で終わらせない。受講者には「自社の業務課題を整理し、どの処理を自動化し、どこにAIを組み込み、どこで人が確認するか」を考えてもらう。

AX・DXワークショップ講座の骨格である「基礎理解、技術適用、評価、提案書作成」を保ちながら、中小企業向けに実装しやすい題材へ置き換える。データサイエンス基礎・応用の内容は、数理やモデルの詳細よりも、業務データをどう整理し、AIや自動化の判断結果をどう評価するかに寄せる。

## 第1回 講師メモ

キーメッセージ:

- DXは大規模システム導入だけではない。
- 中小企業では、転記、確認、通知、報告のような毎日発生する業務を安定させることがDXの入口になる。
- AI自動化は、最初から完全自動化を目指すのではなく、人が確認する工程を残して小さく始める。

講義の流れ:

1. 身近な非効率を例に出す。問い合わせ管理、日報集計、申請受付、請求前チェックなど。
2. As-Is/To-Beの考え方を説明する。
3. 自動化候補を、頻度、作業時間、ミスの影響、ルール化のしやすさ、情報リスクで評価する。
4. 演習で、受講者の業務から候補を3つ出してもらう。

## 第2回 講師メモ

キーメッセージ:

- 自動化の前に、データの置き場と入力ルールを整える。
- Sheetsは小規模業務では台帳、進捗管理、ログ、集計の中心になる。
- Forms、Gmail、Driveはばらばらに使うのではなく、業務フローとしてつなげる。

講義の流れ:

1. Formsで受け付け、Sheetsで管理し、Gmail/Chatで通知し、Driveに証跡を残す流れを示す。
2. Sheetsの列設計を説明する。ID、受付日、担当者、ステータス、期限、対応履歴、最終更新日。
3. Driveのフォルダ構成と権限管理を説明する。
4. 演習で受付・管理シートを設計する。

## 第3回 講師メモ

キーメッセージ:

- GASはGoogle Workspaceの業務部品をつなぐための仕組み。
- 受講者がゼロから全部書ける必要はない。サンプルコードを読み、業務に合わせて修正できる状態を目指す。
- 自動化では、成功時の処理だけでなく、失敗時の通知と復旧が重要。

講義の流れ:

1. GASの画面、実行、承認、ログを見せる。
2. Sheetsからデータを読み、条件に合う行を探す流れを説明する。
3. Gmail通知、Driveファイル作成、Calendarリマインドの例を見せる。
4. トリガーで自動実行する。
5. 演習で、サンプルコードを修正して通知文や対象条件を変える。

## 第4回 講師メモ

キーメッセージ:

- 生成AIは、文章生成、要約、分類、表現調整に強い。
- AIの出力はそのまま業務判断に使わず、確認基準と承認工程をセットにする。
- データサイエンス基礎で扱う「統計解析と機械学習の使い分け」は、本講座では「GASで処理すること、AIに任せること、人が判断すること」の切り分けとして扱う。

講義の流れ:

1. AI、GAS、人の役割分担を説明する。
2. 問い合わせ分類、返信案、議事録要約、日報要約の事例を示す。
3. プロンプトの基本形を説明する。役割、入力、条件、出力形式、禁止事項、確認観点。
4. ハルシネーション、機密情報、著作権、個人情報の注意を入れる。
5. 演習で、AI活用プロンプトと確認チェック項目を作る。

## 第5回 講師メモ

キーメッセージ:

- AI自動化は、要件定義と運用設計がないと現場に定着しない。
- データサイエンス応用で扱う技術選定や評価の考え方は、業務分類、優先順位、通知条件、AI分類結果の評価に応用できる。
- 本番運用では、ログ、エラー通知、権限、手動復旧が重要。

講義の流れ:

1. 要件定義メモの項目を説明する。対象業務、利用者、入力元、出力先、処理条件、例外条件。
2. AI分類や通知条件の評価方法を説明する。誤分類、通知漏れ、過剰通知、対応遅れ。
3. 個人情報、顧客情報、機密情報の扱いを確認する。
4. GASの割り当て制限や大量処理時の注意を説明する。
5. 演習で運用設計書とリスクチェックリストを作る。

## 第6回 講師メモ

キーメッセージ:

- 提案書は、作ったスクリプトの説明ではなく、業務改善の意思決定資料。
- 課題、To-Be、技術方式、KPI、リスク、ロードマップを一貫して説明する。
- レベル3相当では、講師の指示がなくても自分で改善案を組み立てられることを評価する。

講義の流れ:

1. 提案書の構成を説明する。背景、課題、対象業務、To-Be、技術方式、運用、効果、リスク。
2. KPIと効果試算を説明する。削減時間、対応速度、ミス削減、業務品質の安定。
3. ロードマップを作る。試行、本番化、教育、運用、改善。
4. 受講者がミニプロトタイプと提案書を発表する。
5. レビューでは、実現性、リスク対策、現場定着、効果測定を確認する。
