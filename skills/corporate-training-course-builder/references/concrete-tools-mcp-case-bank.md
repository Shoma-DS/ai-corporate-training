# 具体サービス・MCP・事例バンク

スライド案、講師台本、画像生成プロンプト、配布資料に入れる「固有名詞レベルの具体」の正本。抽象的な考え方（入力→処理→出力→確認、委任設計、リスク管理など）を全体像として示したうえで、受講者がその場で具体に落とし込めるように、ここにあるサービス名、MCP名、架空事例を該当スライドへ明記する。

作成時点: 2026年6月。サービスとMCPの提供状況、利用条件、料金、管理者設定は変わり得るため、講座資料には「導入時に最新の公式情報を確認」の注意を必ず添える。参照素材なしでロゴや実在UIを画像生成しない。

確認メモ（2026-06-11）:

- GitHub MCP Server は GitHub 公式リポジトリ `github/github-mcp-server` で、リポジトリ、Issue、PR、Actions などの連携用途が示されている。
- Model Context Protocol は公式ドキュメントで外部ツール・データソース接続の標準として説明されている。MCP連携は便利さだけでなく、権限、トークン、ログ、プロンプトインジェクション対策を同時に扱う。
- Adobe Express は公式サイトでデザイン・写真・動画制作ツールとして提供されている。Adobe Express MCP / Firefly関連MCPなどは、講座内では「Adobe関連MCP候補（要公式確認）」として扱い、公式提供・利用条件・商用利用可否を導入時に確認する。
- ラッコキーワードMCP、kintone MCP、Microsoft 365系MCPなど、提供元や公開状況が変わり得るものは「公開・提供されている場合の候補」として書き、断定しない。

## 使い方ルール

- ツール紹介スライド、ユースケース紹介スライド、連携候補の棚卸しスライド、演習スライドには、このバンクから最低3つの固有名詞を入れる。
- サービス名は必ず目的とセットで書く。「Slack MCPを使う」ではなく「Slack MCPで未対応メンションを抽出し、日次の対応漏れチェックを自動化する」のように業務文脈で書く。
- 事例は実在企業名を使わず、下の架空企業バンクを使う。部署、人数、頻度、件数、分量、所要時間などの数値を必ず入れる。
- 1講座の中で同じ事例企業を使い続けると流れが追いやすい。回をまたいで同じ企業の同じ業務を深掘りする構成を推奨する。
- 提供状況が「要確認」のものは、スライドでは「〜などのMCPが提供・公開されている場合がある。導入時に公式情報を確認」とぼかし、断定しない。

## 架空事例バンク

実在企業名は使わない。以下の架空企業で、数値付きの業務事例を作る。

| 企業 | 業種・規模 | 定番の業務課題（数値付きで使う） |
| --- | --- | --- |
| みどり商事 | 卸売業・従業員60名 | 問い合わせメール1日40件の仕分けと返信下書き、月次売上報告書の作成に毎月8時間、FAX注文の台帳転記ミス月5件 |
| あおば製作所 | 製造業・従業員45名 | 日報30枚の週次集計に毎週3時間、検査記録の転記、設備点検チェックリストの紙運用 |
| ひかり物流 | 物流業・従業員80名 | 配送遅延の問い合わせ対応1日15件、ドライバー日報の集約、請求書照合に月12時間 |
| さくら会計事務所 | 士業・従業員12名 | 顧問先50社の資料受領管理、月次レポート50通の作成、制度改正情報の整理と顧問先向け説明文作成 |
| つばめ商店 | EC小売・従業員8名 | 商品説明文を週20本作成、レビュー返信1日10件、SNS投稿とバナー作成を週5本、検索キーワード調査 |

### 事例の書き方の型

- Before: 「みどり商事では問い合わせメール1日40件を担当者2名が手作業で仕分けし、返信まで平均4時間かかっていた」
- 取り組み: 「Codexで分類スクリプトを試作し、Slack MCPで未対応一覧を毎朝通知。返信下書きはClaudeが作成し、送信前に必ず担当者が確認」
- After（演習上の試算と明示）: 「演習上の試算では、仕分けと下書きで1日約90分の削減。送信ミスは人の確認工程で防止」
- 数値は「演習上の例」「想定例」と明示し、成果保証として書かない。
