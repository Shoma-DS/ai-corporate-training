# AI法人研修 作業ルール

このリポジトリは public です。Google Drive などの社内・提携先資料は参照元として扱い、原本や詳細な営業情報をそのまま転載しないでください。教材、原稿、スライド、配布資料を作るときは、このファイルのルールを優先します。

## ローカルスキル

- このリポジトリ専用のスキルは、リポジトリ直下の `skills/` を正とする。グローバルの `~/.codex/skills/` や各ツール固有の外部スキルより、この直下のスキルを優先して参照する。
- Codex、Claude Code、その他のエージェントは、作業前にこの `AGENTS.md` を読み、依頼内容に合う `skills/*/SKILL.md` がある場合は、そのスキルを読んでから作業する。
- 「講座作成」「講座を作成してください」「研修資料を作って」のような一言依頼では、必ず `skills/corporate-training-course-builder/SKILL.md` を唯一の入口として参照し、同スキルの一気通貫ワークフローに従う。研修資料、講師台本、スライド構成、画像生成プロンプト、スライド画像、配布資料、ワークシート、演習データを作る・直す場合も同じ入口を使う。
- `skills/gws-ai-training-slide-exporter/` や `skills/codex-dynamic-workflows/` は、講座作成スキルから呼び出す下流ヘルパーとして扱う。ユーザーが書き出しだけ、Drive/Slides/Canva連携だけ、またはワークフロー調整だけを明示した場合を除き、これらを講座作成の入口にしない。
- `skills/course-pamphlet-html-pdf/` は、パンフレットHTML/PDF作成の下流ヘルパーとして扱う。新規パンフレットはMarkdownではなく `パンフレット.html` を正として作り、提出用の `パンフレット.pdf` まで生成する。既存の `パンフレット原稿.md` や `パンフレット.md` は移行元として扱い、内容を変えた場合は同スキルのスクリプトでHTMLとPDFへ反映する。
- このリポジトリで今後スキルを作る・更新する場合は、独立した講座作成スキルを増やさず、原則として `skills/corporate-training-course-builder/` の参照ファイル、チェックリスト、スクリプト、または明示的な下流ヘルパーとして統合する。新しい手順は講座作成ワークフローから到達できるようにし、作っただけで使われないスキルにしない。
- 同スキル内で詳細チェックリストが必要な場合は、`skills/corporate-training-course-builder/references/session-production-workflow.md` を参照する。
- `講座/AX・DXワークショップ講座/` のスライド案、講師台本、画像生成プロンプト、スライド画像を作る・直す場合は、既存の合格基準PDF `AX・DXワークショップ講座_第1回 .pdf` と `AX・DXワークショップ講座_第2回.pdf` を必ず参照する。参照するのはネタや文言の転載ではなく、スライド1枚ごとの情報量、具体度、So What型ヘッドライン、比較表・プロセス図・事例・リスク対策の置き方とする。原本PDFは public repo に保存しない。
- AX・DX講座の構成、章立て、総スライド枚数は講座ごとの狙いに合わせて変えてよい。ただし1枚1枚は薄くしない。各スライドはタイトルだけで済ませず、原則として結論型ヘッドライン、3〜6個の具体ブロック、比較表・プロセス・チェックリスト・事例・判断基準のいずれかを持たせ、参照PDFと同等の説明密度にする。
- スライド画像、講座ビジュアル、画像素材を GPT image 2 で作る場合、またはユーザーが「上から処理ではなく1枚まるごと生成」と指定した場合も、`skills/corporate-training-course-builder/SKILL.md` の画像生成ルールに従う。
- スライド画像の「再生成」「作り直し」「GPT image 2」指定では、既存PNGの上から文字や要素を重ねて直さず、SVG/HTML/CSS/canvas/ローカル変換を中間に挟まず、公式ロゴなどの参照素材を読み込んだうえで1枚の完成画像として生成し直す。
- `スライド画像/Sxx.png` は、GPT image 2 / built-in `image_gen` で1枚の完成ビットマップとして生成した画像、または規約確認済みの公式スクリーンショット・公式素材だけを完成物として認める。SVG、HTML/CSS、canvas、ブラウザスクリーンショット、PIL/Pillow、ImageMagick、PDF/PPTX書き出し、ローカルラスタライズ、ローカル合成、テキスト後載せで作ったPNGを `スライド画像/` の完成物として保存しない。
- 画像生成できていないスライドは「未生成」と記録し、仮画像・プレースホルダー・ローカル描画画像で生成済み扱いにしない。スクリプトは `スライド案.md`、`講師台本.md`、`画像生成プロンプト.md`、配布資料、演習データ、PPTX組み立てまでに限定し、`スライド画像/Sxx.png` を自動描画しない。
- Canvaで複数ページの研修プレゼンを作る場合は、まずCanva MCP/APIまたはPPTX取り込みで、`スライド画像/Sxx.png` が1ページ目から最終ページまで順番に並ぶ画像ベースのプレゼンテーションを作る。いきなりブラウザ操作で全ページを1枚ずつ作らない。
- CanvaのMagic Layersを使う場合は、作成済みの複数ページプレゼンをブラウザで開き、対象ページを1枚ずつ適用、数秒待機、文字化け・文言変化・レイアウト崩れを検証する。失敗した場合はCommand+Z、戻る、または履歴操作で適用前に戻して再試行し、成功・失敗・再試行回数を `非公開/Canva/` に記録する。
- Canva取り込みやMagic Layersで品質を保ちやすい設定・工夫がある場合は、エージェント側で積極的に採用する。同じページでMagic Layersが3回以上失敗する場合は、それ以上変換せず元画像のまま残す。失敗から得た確認観点は `skills/gws-ai-training-slide-exporter/references/canva-quality-checklist.md` に追記し、次回以降の重点チェックに使う。
- Kimi WebBridgeなど実ブラウザ操作はトークン消費が大きいため、サブエージェントとモデル切替が使える環境では、ブラウザ操作担当を低コストの `GPT-5.3 Codex Spark` 相当、司令塔・最終判断・文書統合を高精度モデル、画像生成やGPT image 2が必要な作業を画像生成対応の高精度モデルに分ける。モデル指定ができない環境では、同じ役割分担を手順として分離し、ブラウザ操作のスクリーンショット取得と検証回数を最小化する。
- ローカルスキルやフロントマターの確認は Ruby ではなく Python を使う。確認が必要な場合は `python3 scripts/validate_local_skills.py` を実行する。

## 参照元

- 主参照フォルダ: ユーザー指定の法人研修 Google Drive フォルダ
- ローカル取得先: `非公開/法人研修Drive原本/`
- 抽出テキスト: `非公開/法人研修Drive抽出テキスト/`
- `非公開/` は Git 管理外。会社資料、提携先資料、動画リンク、価格、連絡先、申請書類の実物は public repo にコミットしない。

新しい資料を作る前に、対象講座と近い既存資料を Drive または `非公開/` で確認する。公開ファイルに残すのは、再利用できる制作ルール、抽象化した構成、出典メモまでにする。

## Git運用

- 原則として `main` に直接コミットして push する。作業ブランチや PR は、ユーザーが明示した場合、または破壊的・大規模な変更で確認導線が必要な場合に限る。
- コミットメッセージは日本語で書く。1行目は小学生にもわかる短い要約にし、空行の後に3〜5行で「どこを変えたか」「何をしたか」「何がよくなるか」を書く。
- push 前に `非公開/`、出典PDF、`.DS_Store`、個人情報や社内固有情報がステージされていないことを確認する。

## 基本方針

- 研修は「ツール紹介」ではなく、業務課題の解決、職務遂行力の向上、DX推進、継続運用までを扱う。
- 受講者が明日から使える実務アウトプットを持ち帰れる構成にする。
- 各講座は、課題提起、対象者、受講後にできること、研修の強み、提供方式、標準学習時間、カリキュラムの順で整理する。
- 生成AIは単体ツールとしてではなく、思考整理、情報整理、文書作成、分析、資料作成、業務フロー化、定着運用の一連の流れで扱う。
- 初心者向け、中級者向け、高度デジタル人材向けを混同しない。前提知識、到達目標、演習の難度を明確に分ける。

## 講座区分

### リスキリングコース

- 原則として eラーニング、LMS による受講状況と受講時間の記録を前提に書く。
- 標準構成は 6回、各120分、合計約12時間、2か月以内を基本形にする。
- 主な読者は、生成AI、DX、データ活用を実務に取り入れたい企業担当者。
- 初心者向けは、専門用語を抑え、日常業務への適用例を多くする。
- 応用向けは、AIの使い分け、業務フロー設計、Copilot、NotebookLM、Gamma、Dify、RAG などの実務適用まで扱う。

### 人への投資促進コース 高度デジタル人材

- ITSS+ レベル3相当の「独力で業務課題の整理、要件定義、解決策の設計、提案ができる」状態を意識する。
- 単なる操作研修ではなく、現状把握、データ棚卸し、ユースケース選定、要件定義、設計、評価、提案書作成までを扱う。
- 成果物は、AX/DX企画提案書、RAG設計案、AIアプリ、データ活用プランなど、実務で提案・推進に使える形にする。
- 受講者は、生成AIやデータ活用の基礎を持ち、部署や事業の改善を主体的に進める立場を想定する。

### 人への投資促進コース 定額制訓練

- 年間の定額制、選択受講、必要テーマだけの受講、部署ごとの組み合わせ、段階的なスキルアップを訴求軸にする。
- 営業、事務、企画、管理、広報、マーケティング、現場スタッフなど幅広い職種を対象にできる。
- 1テーマ完結型にし、忙しい中小企業でも業務の合間に学べる設計にする。
- 全体は、AI活用による業務遂行力向上と、AIツール活用による実務高度化に分けて整理すると読みやすい。

## 資料構成

パンフレット、講座紹介、スライド構成案は次の順にする。パンフレットは `講座名/全体/パンフレット.html` として作り、提出用に `講座名/全体/パンフレット.pdf` も生成する。Markdownのパンフレット原稿を新規作成の正本にしない。

1. タイトルと一言訴求
2. 研修の目的
3. よくある悩み、導入背景
4. この研修で解決すること
5. 受講後にできること
6. 研修の強み
7. 対象者、推奨される知識・経験
8. 提供方式、学習時間、受講管理
9. カリキュラム表
10. 演習、成果物、業務への持ち帰り
11. 注意事項、申込や運用に関する確認事項

各回のカリキュラムは、回数、テーマ、時間、主な内容、成果物または学習目標をそろえる。6回構成では、基礎理解から始め、実務適用、ケーススタディ、最後に自社・自部署への実装計画へ進める。

## 表現ルール

- 文章は法人向けに、実務的でわかりやすく書く。
- 「明日から使える」「業務に組み込む」「自走できる」「継続運用」「属人化の解消」「業務品質の安定」「提案力向上」など、既存資料と相性のよい語彙を使う。
- 誇大な成果保証は避ける。数値改善例を書く場合は、事例、想定例、演習上の例のどれかを明示する。
- AIツール名は目的とセットで書く。例: Claude は思考整理、NotebookLM は情報整理、Gamma は資料作成、Copilot は Microsoft 365 業務、Dify はAIアプリ開発。
- 「AIを使う」だけで終わらせず、入力、処理、出力、確認、運用改善まで書く。
- 価格、連絡先、会社名、助成率、申請期限など変わり得る情報は、公開資料に固定せず、最新の参照元で確認する。

## 助成金関連の扱い

- 助成金に関する説明は、必ず最新の公的情報と社内確認を前提にする。
- 資料では、対象者、訓練時間、OFF-JT、LMS記録、修了証、支給申請、計画申請などの確認観点を漏らさない。
- 受給可否を断定しない。制度要件、提出期限、審査方法、助成率、上限額は変更され得るため、作成時点の日付と確認元を残す。
- 支給申請や受講ログに関わる個人情報、雇用保険番号、署名、口座情報、会社固有情報は絶対に public repo に保存しない。

## AI・情報管理の注意

- 生成AI活用資料では、情報漏洩、著作権、ハルシネーション、機密情報の入力禁止、出力確認を必ず扱う。
- 企業固有データを扱う演習では、ダミーデータまたは匿名化データを使う。
- RAG、NotebookLM、Dify など社内資料を読み込む内容では、対象データ、権限、更新、根拠提示、ログ確認、改善方法をセットで説明する。
- 顧客情報、社員情報、未公開資料、契約情報は `非公開/` のみで扱い、公開成果物には抽象化した説明だけを残す。

## スライド素材の扱い

- サービス紹介、ツール紹介、画面操作、作業風景を伝えるスライドでは、すべてを抽象図解で表さない。公式画面のスクリーンショット、操作画面、作業風景の写真や画像を使う方が理解しやすい場合は、図解より実画面・実素材を優先する。
- サービスを紹介する場合は、必ず公式サイト、公式ブランドガイドライン、公式プレスキットなどからロゴを検索し、公式ロゴ素材をダウンロードして使う。画像生成AIに実在サービスのロゴを描かせない。
- パンフレットの表紙、中扉、見出し画像など、画像自体に講座名やサービス要素を入れる場合は、HTML/CSSで画像の上に文字やロゴを後載せしたような作りにしない。GPT image 2 などで1枚の完成画像として生成し、公式ロゴが必要な場合は取得済みの公式ロゴ画像を参照画像として読み込ませ、講座ビジュアルの一部として自然に取り込む。プロンプトだけで実在ロゴを想像生成させない。
- 公式ロゴやスクリーンショットを使う場合は、利用規約、ブランドガイドライン、著作権、商標表記、出典URLを確認する。公式ロゴはリポジトリ直下の `素材/ロゴ/` に集約し、取得元メモも同じ場所に残す。講座ごとの調査メモは講座名フォルダ直下の `全体/調査/` に残す。
- 画面スクリーンショットには、個人情報、顧客情報、社内情報、メールアドレス、ファイル名、ワークスペース名、APIキー、契約情報が写り込まないようにする。必要に応じてダミー環境または公式公開画像を使う。
- 画像生成プロンプトでは、プロンプトだけで実在ロゴ、実在サービス画面、実在UIの忠実な再現を指示しない。必要な場合は、公式素材を別途取得し、参照画像として読み込ませて生成する。ロゴの正確性が保てない場合や規約上の懸念がある場合は、生成画像内に無理に入れず、公式素材として別管理する。

## 成果物の保存場所

- 原稿: `原稿/`
- スライド構成案: `スライド/`
- 配布資料、ワークシート: `配布資料/`
- 講座全体の概要、詳細シラバス、全回ワークシート、回別一覧、パンフレットHTML/PDF: `講座名/全体/`
- 講座全体の出典メモ、調査メモ: `講座名/全体/調査/`
- 公開可能な図版、素材: `素材/`
- 会社資料、Drive原本、動画台本、非公開素材: `非公開/`

講座を回別に制作する場合、各回フォルダ直下には `スクリーンショット/`、`配布資料/`、`演習データ/`、`スライド画像/` を作る。画面共有や操作説明で使う画像は `スクリーンショット/` に置き、`素材/作業風景/` のような別フォルダは作らない。講座全体で使う概要、シラバス、全回資料、回別一覧、パンフレット、調査メモは講座名フォルダ直下の `全体/`（調査メモは `全体/調査/`）、複数講座で使う公式ロゴはリポジトリ直下の `素材/ロゴ/` を参照する。

## 作成前チェック

- 対象講座区分は明確か。
- 受講者の前提知識と到達目標は合っているか。
- 業務課題から始まり、実務アウトプットに着地しているか。
- LMS、学習時間、受講ログ、修了確認など運用要件を落としていないか。
- 生成AIのリスクと確認手順を入れているか。
- public repo に載せてはいけない情報を含んでいないか。

## ブラウザ操作のモデル分担

- Kimi WebBridge、agent-browser、Canva、実サイト確認などのブラウザ操作はトークン消費が大きいため、サブエージェントとモデル指定が使える環境では `codex-5.3spark` / `GPT-5.3 Codex Spark` 相当をブラウザ操作担当に優先指定する。
- Spark担当はクリック、待機、ページ移動、スクリーンショット取得、単純な表示確認、反復的なブラウザ検証を担当し、Spark利用枠を積極的に消費する。
- メインエージェントまたは高精度モデルは、作業計画、合否判断、公開可否、最終文言、設計判断、ファイル統合を担当する。ブラウザ操作を高精度モデル単体で長時間抱え込まない。
- 画像生成、複雑な設計判断、最終品質判断が必要な作業は、Spark担当ではなくメインエージェントまたは適切な高精度・画像対応モデルに戻す。
- git status/diff/add/commit/push などの定型的なGit操作も、サブエージェントとモデル指定が使える環境では `codex-5.3spark` / `GPT-5.3 Codex Spark` 相当に優先して任せ、メインエージェントは差分の妥当性、コミット範囲、公開不可情報の混入有無、最終判断を担当する。
- モデル指定やサブエージェント呼び出しができない環境では、同じ役割分担を手順として分離し、スクリーンショット回数、snapshot回数、ページ移動回数を最小化する。
