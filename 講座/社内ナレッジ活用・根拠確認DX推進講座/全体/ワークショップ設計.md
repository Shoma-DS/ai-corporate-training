# ワークショップ設計

## 設計方針

この講座は、NotebookLMの機能紹介ではなく、業務資料を根拠付きで読み解き、現場の判断と文書作成に使える状態へ変えるワークショップとして設計する。

録画動画形式でも成立するよう、講師は実演画面を見せ、受講者には「ここで動画を一時停止して、○分ほど取り組んでください。取り組めたら再生してください。」と案内する。発表、相互レビュー、ライブ討議は行わず、講師記入例との自己レビューに置き換える。

## 学習者が作る最終成果物

1. 業務課題棚卸しシート
2. 情報源棚卸し表
3. 根拠付きFAQ
4. 会議アクション一覧
5. NotebookLM業務活用設計シート
6. ナレッジ運用ミニ提案書

## ワーク1: 業務課題を選ぶ

目的:

NotebookLMを使う前に、どの業務の情報探索、確認、要約、説明、共有を改善したいかを決める。

手順:

1. 日常業務で「資料を探す」「過去の経緯を確認する」「マニュアルを読む」「会議内容を整理する」場面を書き出す。
2. 時間がかかる、ミスが起きる、属人化している、確認先が曖昧、の観点で困りごとを整理する。
3. NotebookLMに向くテーマを1つ選ぶ。

判断基準:

- 複数資料を横断して確認する必要がある。
- 根拠を示しながら回答したい。
- FAQ、説明文、比較表、アクション一覧に変換したい。
- 機密情報や個人情報を含まない、または安全にダミー化できる。

## ワーク2: 情報源を設計する

目的:

NotebookLMに入れる資料を選び、入れてはいけない資料を除外する。

手順:

1. 資料名、形式、目的、更新日、責任者、機密区分を記入する。
2. 取り込み可否を、公開情報、社内限定、個人情報あり、契約情報あり、未公開情報あり、の観点で判断する。
3. NotebookLMに入れる前に必要な匿名化、要約化、抜粋化、承認を記入する。

確認観点:

- 最新版か。
- 権限があるか。
- 個人情報、顧客情報、契約情報、価格、連絡先が含まれていないか。
- 著作権や利用規約上、AI入力が許容されるか。
- Driveから取り込む場合、NotebookLM側にコピーされる前提を理解しているか。

## ワーク3: 根拠付きで質問する

目的:

NotebookLMの回答を業務に使える形へ変換する。

質問例:

- この資料群から、問い合わせ対応で最初に確認すべき項目を5つ挙げてください。各項目に根拠となる資料名を付けてください。
- 新人向けに、返品対応の手順を300字で説明してください。例外対応と確認先も分けてください。
- 会議メモから、担当者、期限、決定事項、未決事項を表にしてください。根拠が曖昧なものは「要確認」としてください。
- 提案資料とFAQを比較し、顧客に説明するときに誤解されやすい点を列挙してください。

出力確認:

- 根拠が資料内にあるか。
- 古い資料と新しい資料が混在していないか。
- 断定してはいけないことを断定していないか。
- 実務で使う前に確認すべき担当者、原典、追加資料が明示されているか。

## ワーク4: 業務アウトプットへ変換する

目的:

要約で終わらせず、現場で使う文書に変換する。

作成するアウトプット:

- 根拠付きFAQ
- 顧客向け説明文案
- 社内確認チェックリスト
- 会議アクション一覧
- 新人向け学習メモ
- 運用改善の提案メモ

レビュー観点:

- 読み手に合わせた言葉になっているか。
- 重要な制約や例外が抜けていないか。
- 最終判断者、確認先、更新日が残っているか。
- AI出力と原典確認の役割分担が明確か。

## ワーク5: 継続運用を設計する

目的:

個人の便利技で終わらせず、部署で使い続けられるルールにする。

記入項目:

- 対象業務
- ノートブック名
- 入れる資料
- 入れない資料
- 更新頻度
- 更新担当
- レビュー担当
- 共有範囲
- 誤回答を見つけたときの対応
- 削除・停止の条件
- 導入後に見るKPI

KPI例:

- 情報探索時間の削減
- 問い合わせ一次回答の作成時間短縮
- 会議後アクション整理の漏れ削減
- マニュアル更新時の現場周知スピード向上
- 新人の確認先問い合わせ件数の低減
