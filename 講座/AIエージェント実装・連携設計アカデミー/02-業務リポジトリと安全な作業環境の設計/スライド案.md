## 高密度スライド詳細

この節は `画像生成プロンプト.md` と同期する提出前確認用の正本です。各スライドは、講師台本を読まなくても学習内容、受講者アクション、成果物、確認観点が分かる密度にしています。

### S01 業務リポジトリと安全な作業環境の設計

**ヘッドライン:** AIに作業させる前に、見せる情報・触らせるファイル・承認が必要な操作を決める
**テンプレートID:** `isometric-corporate-clean`
**図解パターン:** `scene-environment`
**素材・画面:** 図解
**主な内容:** 安全な作業範囲、公開不可情報、承認ルール
**推奨レイアウト:** 右側に小さな業務場面、左側にタイトル・ヘッドライン・3枚の要点カードを置く。実在UIやロゴは描かない。
**参照素材・スクリーンショット:** 生成図解を基本にする。実在ロゴ・実在UIは参照素材がある場合のみ使い、参照素材なしではサービス名を文字カードで示す。

**内容ブロック:**
- 判断すること
  - 安全な作業範囲、公開不可情報、承認ルールを、業務課題、入力情報、AI/ツールの役割、人の確認、成果物、停止条件の観点で読む。
  - このスライドは `業務改善リポジトリ設計、AGENTS.md案、ダミーデータ方針、作業安全チェック` へ接続する。
- 具体サービス・MCP候補
  - Codex, GitHub MCP, Filesystem MCP, Playwright MCP を目的とセットで扱う。提供状況、利用条件、料金、管理者設定は導入時に最新の公式情報を確認する。
  - 実在ロゴや実在画面は想像生成せず、参照素材がない場合は文字カードで示す。
- 数値付き架空事例
  - みどり商事（卸売業・従業員60名）では、問い合わせメール1日40件、月次売上報告8時間、FAX注文の転記ミス月5件を題材に、実データではなくダミーCSVと架空部署名だけで作業環境を作る。
  - 数値は演習上の例または想定例であり、成果保証ではない。
- 受講者アクション・確認観点
  - 安全な作業範囲、公開不可情報、承認ルールを自部署の業務に置き換え、入力、処理、出力、人の確認、ログ/保存先、停止条件を書けるか確認する。

  - リスク確認: 実データ、顧客情報、社員情報、価格、連絡先、契約情報、認証情報、未公開資料は使わない。

**成果物・確認観点:** 業務改善リポジトリ設計、AGENTS.md案、ダミーデータ方針、作業安全チェックへ反映できる形で、成果物名、確認者、レビュー基準、停止条件を明記する。
