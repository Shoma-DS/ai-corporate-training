# 全回講師用メモ

講座名: AIエージェント実装・連携設計アカデミー

## 共通方針

- 録画動画によるeラーニングとして進行する。
- ワークは「ここで動画を一時停止して、○分ほど取り組んでください。取り組めたら再生してください。」と案内する。
- ライブ発表、チャット共有、相互レビューは行わない。代わりに自己レビューと講師の記入例との比較を入れる。
- Codex、MCP、接続先SaaSの最新仕様、利用可否、料金、利用上限、管理者設定は固定しない。
- 実演はサンプルリポジトリとダミーデータで行う。
- 画面共有で実在サービスを見せる場合は、公式公開画面またはダミー環境を使い、個人情報、顧客情報、認証キー、契約情報を写さない。

## たとえ話

- Codex: 仕様書を読んで作業する補助メンバー。指示が曖昧だと、違う方向へ進むため、依頼書と確認条件が必要。
- MCP: AIエージェント用の接続口。工具箱を渡すのではなく、使ってよい工具、使ってよい場所、使う前の確認を決める。
- AGENTS.md: 現場の作業手順書。新人に「この倉庫で何を触ってよいか」を伝える掲示板。
- サンドボックス: 作業エリアの柵。柵の中では素早く作業し、柵の外に出るときは承認する。
- Resources: 参照してよい資料棚。
- Tools: 実際に手を動かす道具。書き込みや外部送信は特に注意。
- Prompts: 毎回同じ品質で作業するための作業指示書。

## 回別の注意

| 回 | 注意点 |
| ---: | --- |
| 1 | CodexやMCPの機能説明に寄りすぎない。業務課題とタスクブリーフ作成に集中する。 |
| 2 | Gitやターミナルで脱落させない。目的は高度な開発ではなく、作業範囲と安全ルールを整理することだと強調する。 |
| 3 | 動くものを作るだけで終えない。差分確認、テスト、過剰実装の抑止、人の確認を必ず入れる。 |
| 4 | MCPは「何でもつなげる」ではなく「つながない判断」も設計に含める。秘密情報は見せない。 |
| 5 | セキュリティを恐怖訴求にしない。実務で確認できるリスク、承認、ログ、復旧へ落とす。 |
| 6 | 便利だった感想ではなく、導入判断者が見たい課題、効果、リスク、運用、ロードマップへまとめる。 |
