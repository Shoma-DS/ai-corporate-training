# 第4回 スライド構成案: MCPで社内ツールとデータをつなぐ設計

対象講座: AIエージェント実装・連携設計アカデミー

## 設計方針

- 採用テンプレート: `isometric-corporate-clean`
- 120分の録画eラーニングとして、MCPを「AIエージェントの安全な接続設計」として扱う。
- MCPの技術用語は、Resources、Prompts、Tools、Host/Client/Server、認証、承認、ログに絞って、業務設計へ落とす。
- 実演はダミー設定と棚卸し表で行う。実在SaaS、実トークン、社内URL、顧客データは扱わない。

### S01 MCPで社内ツールとデータをつなぐ設計

**ヘッドライン:** MCPで何を参照し、何を実行させ、どこに承認を置くかを設計する
**テンプレートID:** `isometric-corporate-clean`
**図解パターン:** `scene-environment`
**素材・画面:** 図解
**主な内容:** Resources、Prompts、Tools、認証・承認・ログの設計
**推奨レイアウト:** 右側に小さな業務場面、左側にタイトル・ヘッドライン・3枚の要点カードを置く。実在UIやロゴは描かない。
**参照素材・スクリーンショット:** 生成図解を基本にする。実在ロゴ・実在UIは参照素材がある場合のみ使い、参照素材なしではサービス名を文字カードで示す。

**内容ブロック:**
- 判断すること
  - Resources、Prompts、Tools、認証・承認・ログの設計を、業務課題、入力情報、AI/ツールの役割、人の確認、成果物、停止条件の観点で読む。
  - このスライドは `MCP連携設計書、ツール棚卸し、MCP設定ドラフト、接続しない判断リスト` へ接続する。
- 具体サービス・MCP候補
  - GitHub MCP, Slack MCP, Notion MCP, Google Drive MCP を目的とセットで扱う。提供状況、利用条件、料金、管理者設定は導入時に最新の公式情報を確認する。
  - 実在ロゴや実在画面は想像生成せず、参照素材がない場合は文字カードで示す。
- 数値付き架空事例
  - ひかり物流（物流業・従業員80名）では、配送遅延問い合わせ15件/日、ドライバー日報集約、請求照合12時間/月を、Slack・Notion・Drive・DB連携候補に分けて設計する。
  - 数値は演習上の例または想定例であり、成果保証ではない。
- 受講者アクション・確認観点
  - Resources、Prompts、Tools、認証・承認・ログの設計を自部署の業務に置き換え、入力、処理、出力、人の確認、ログ/保存先、停止条件を書けるか確認する。
  - 自己レビュー: ツール名だけになっていないか、入力、処理、出力、確認者、ログ/保存先、停止条件が書けているか確認する。
  - リスク確認: 実データ、顧客情報、社員情報、価格、連絡先、契約情報、認証情報、未公開資料は使わない。

**成果物・確認観点:** MCP連携設計書、ツール棚卸し、MCP設定ドラフト、接続しない判断リストへ反映できる形で、成果物名、確認者、レビュー基準、停止条件を明記する。
