- 採用テンプレート: `isometric-corporate-clean`

## 高密度スライド詳細

この節は `画像生成プロンプト.md` と同期する提出前確認用の正本です。各スライドは、講師台本を読まなくても学習内容、受講者アクション、成果物、確認観点が分かる密度にしています。

### S01 評価・権限・運用設計と定着化

**ヘッドライン:** AIエージェントを継続運用できるように評価・権限・復旧・教育を設計する
**テンプレートID:** `isometric-corporate-clean`
**図解パターン:** `scene-environment`
**素材・画面:** 図解
**主な内容:** 評価、権限、ログ、復旧、教育まで含む運用設計
**推奨レイアウト:** 右側に小さな業務場面、左側にタイトル・ヘッドライン・3枚の要点カードを置く。実在UIやロゴは描かない。
**参照素材・スクリーンショット:** 生成図解を基本にする。実在ロゴ・実在UIは参照素材がある場合のみ使い、参照素材なしではサービス名を文字カードで示す。

**内容ブロック:**
- 判断すること
  - 評価、権限、ログ、復旧、教育まで含む運用設計を、業務課題、入力情報、AI/ツールの役割、人の確認、成果物、停止条件の観点で読む。
  - このスライドは `運用設計書、リスクチェックリスト、テスト観点表、手動復旧手順` へ接続する。
- 具体サービス・MCP候補
  - Codex, Sentry MCP, Slack MCP, GitHub MCP を目的とセットで扱う。提供状況、利用条件、料金、管理者設定は導入時に最新の公式情報を確認する。
  - 実在ロゴや実在画面は想像生成せず、参照素材がない場合は文字カードで示す。
- 数値付き架空事例
  - あおば製作所（製造業・従業員45名）では、日報30枚/週、検査記録転記、点検チェックリスト紙運用を題材に、誤実行・権限過多・復旧手順を確認する。
  - 数値は演習上の例または想定例であり、成果保証ではない。
- 受講者アクション・確認観点
  - 評価、権限、ログ、復旧、教育まで含む運用設計を自部署の業務に置き換え、入力、処理、出力、人の確認、ログ/保存先、停止条件を書けるか確認する。
  - 自己レビュー: ツール名だけになっていないか、入力、処理、出力、確認者、ログ/保存先、停止条件が書けているか確認する。
  - リスク確認: 実データ、顧客情報、社員情報、価格、連絡先、契約情報、認証情報、未公開資料は使わない。

**成果物・確認観点:** 運用設計書、リスクチェックリスト、テスト観点表、手動復旧手順へ反映できる形で、成果物名、確認者、レビュー基準、停止条件を明記する。
