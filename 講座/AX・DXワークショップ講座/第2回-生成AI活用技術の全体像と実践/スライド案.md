### S05 汎用AIツール利用は通過点にすぎない
**ヘッドライン:** 競争有意性は「汎用ツールを使う（30%）」ではなく、「自社ナレッジを磨き込んだ自社AI（100%）」にある

**段階①｜汎用AIツールの活用（30%・現在多数の企業）**
- ChatGPT等の汎用ツールで社内業務効率化・アイデア出し・文章生成
- **問題:** 競合他社も同じツールを使っている → コモディティ化 → Excelを使うのと同じ水準

**段階②｜"自社AI"への進化（60%・現在〜近未来）**
- 社内独自データ（顧客情報・技術情報・業務ログ・過去事例）をAIに連携・学習
- Fine-tuning / RAGで「自社専用モデル・システム」を構築
- 「模倣困難な知的資産」の形成が始まる

**段階③｜持続的競争優位性の確立（100%・目標）**
- 顧客体験の革新・新ビジネス創出・差別化された価値提供
- AIが業務プロセスに深く組み込まれ、人と組織の能力を継続的に増幅

**今、取り組むべき理由:**
- AI活用の先行者は「学習データ蓄積」「ノウハウ蓄積」で先行を広げていく
- 1〜2年の遅れが数年分の競争優位格差になる可能性

- **図解パターン:** process-flow

### S06 従来の生成AIと現在の生成AIの進化
**ヘッドライン:** 現在の生成AIは「文章作成」を超え、外部ツール・APIと連携して自律的にタスクを実行できる段階に進化している

| 観点 | 従来の生成AI（〜2022年） | 現在の生成AI（2023年〜） |
|---|---|---|
| 主な機能 | 文章作成・要約・翻訳・コード生成 | 上記＋外部ツール連携・API呼び出し・自律的な判断 |
| 利用形態 | チャット形式（1問1答） | エージェント型（複数ステップを自律実行） |
| システム連携 | 基本なし | 他システム・データベース・メール等と連携可能 |
| ナレッジ活用 | 学習データのみ（カットオフあり） | RAGで最新情報・自社情報を参照して回答 |
| 適用範囲 | 個人の作業効率化 | 組織のビジネスプロセス自動化・改善 |

**変化の本質:**
- 「チャットで答えを得る」→「業務プロセスの中でAIが動き続ける」
- 個人の生産性ツールから、組織の業務インフラへ

- **図解パターン:** before-after-transformation

### S07 社内情報とAIの壁：ナレッジ連携が競争優位性を生む
**ヘッドライン:** 汎用AIは「一般知識」しか持たない。自社固有の文脈に沿った答えを得るには、社内ナレッジとの連携が必須

**ツール単体での利用の限界:**

❌ **例1: 復旧手順の質問**
- 入力: 「○○システムの障害時の復旧手順を教えて」
- 出力: 「一般的な回答・他社事例」→ 自社固有手順とは異なる → 現場で使えない

❌ **例2: 顧客返信文の作成**
- 入力: 「○○様への返信文を書いて」
- 出力: テンプレート文面 → 過去の経緯・個別事情を踏まえられない → 不適切な文章

**社内ナレッジとの連携後:**

✅ **例1の改善: 自社手順書をRAGで参照**
- 入力: 同じ質問
- 出力: 自社手順書に基づく正確な回答（根拠ページ付き）

✅ **例2の改善: 顧客履歴と連携**
- 入力: 同じ質問
- 出力: 顧客との経緯・文脈を踏まえた適切な文面案

**本当に欲しい答え = 自社の文脈に基づく回答**

### S08 自社データ連携の2つの入口
**ヘッドライン:** 開発なしで始められる「ノートブック型」と、日常業務に統合される「ストレージ自動連携型」─課題に合わせて選択する

**ノートブック型（その場限り・手動選定）:**
- 代表ツール: Google NotebookLM / Copilot Notebook
- 仕組み: ユーザーがドキュメントをアップロード → そのセッション内でAIが参照
- 強み: 開発不要・すぐに使える / 参照範囲が明確
- 弱み: その場限り / 大量文書の常時参照には不向き
- **適用シーン:** 特定プロジェクトの資料調査・一時的な情報整理

**ストレージ自動連携型（常時・全データ参照）:**
- 代表ツール: Microsoft 365 Copilot / Gemini for Google Workspace
- 仕組み: クラウドストレージに保存された全ファイルをAIが自動参照
- 強み: 日常業務に統合 / 権限内の全データを参照可能 / アップロード不要
- 弱み: 管理者設定が必要 / データの整理・権限設計が重要
- **適用シーン:** 全社的なAI活用 / メール・ドキュメント・会議との統合

**どちらも「自社データ・ナレッジ活用の入口」として有効**

- **図解パターン:** comparison-contrast

### S09 競争優位性は「自社アセット × AI」にある
**ヘッドライン:** 汎用生成AIはコモディティ。「自社独自のナレッジ・データ」を掛け合わせることで、初めて他社に真似できない優位性が生まれる

**汎用モデル（生成AI）単体の限界:**
- Excelを使う技術が同じなら、出力物の質に差はつかない
- ChatGPT等の汎用ツールは誰でも使える → コモディティ化

**自社独自の資産（ナレッジ・データ）の価値:**
- 過去の記録（商談履歴・クレーム対応記録・プロジェクト事例）
- 現場の暗黙知（熟練担当者のノウハウ・判断基準）
- 顧客対応データ（問い合わせパターン・解決策・満足度）
- 自社製品・サービスの詳細仕様

**組み合わせで生まれるもの:**
- 汎用モデル × 自社独自資産 = **自社AIによる競争優位性**
- 模倣困難な「知的資産」の形成
- 顧客体験の革新・新ビジネス創出・差別化された価値提供

**重要な示唆:** 今すぐ「自社固有のデータ」を整理・蓄積を始めることが最大の競争優位への投資

### S10 ハルシネーションの構造的背景と3段階の知識ギャップ
**ヘッドライン:** LLMは「分からない」と黙るより「推測して答える」ほうが評価されるため構造的にハルシネーションが発生し、生成AIが「知らない」2つのギャップも存在する

**ハルシネーションの構造:**

| 状態 | LLMの選択 | 理由 |
|---|---|---|
| 「分からない」（沈黙） | 0点の可能性 | 答えを出せないと評価が下がる |
| 「たぶんこう…」（推測） | 正解なら高得点 | もっともらしい文章なら評価される |

→ LLMは統計的に「推測して答える」インセンティブが働く

**生成AIが「知らない」2つのギャップ:**

① **Time Gap（学習カットオフ後の情報）**
- 例: ニュース・株価・法改正・製品アップデート
- 対策: Web検索との連携 / 最新情報はユーザーが補完

② **Access Gap（未公開の自社情報）**
- 例: 社内規定・顧客情報・自社ノウハウ・未公開データ
- 対策: RAG（社内ドキュメントをAIが検索できる形で保持）

**3段階の管理:**
1. ハルシネーション対策（プロンプト設計・出力確認ルール）
2. Time Gap補完（Web検索連携・情報補完）
3. Access Gap補完（社内ナレッジRAG構築）

- **図解パターン:** governance-risk

### S12 生成AI活用5レイヤー全体像
**ヘッドライン:** 「コスト・複雑さ」と「自社ナレッジの必要性」の2軸で5レイヤーを整理すると、段階的な高度化の道筋が見える

| レイヤー | 名称 | コスト・複雑さ | 自社ナレッジ必要性 | 適した状況 |
|---|---|---|---|---|
| **L1** | 汎用AIの「ツール利用」 | 低 | 不要 | まず試したい / 個人効率化 |
| **L2** | 汎用AIの「API利用」 | 低〜中 | 少量でも可 | 自動化・システム組み込みしたい |
| **L3** | RAG（検索拡張生成） | 中 | 必要（社内文書） | 自社情報を参照して回答させたい |
| **L4** | AIエージェント | 高 | 大量に必要な場合も | 複数タスクを自律実行させたい |
| **L5** | ファインチューニング | 高 | 大量（高品質データ） | 固有の文体・専門知識を組み込みたい |

**重要な注意:**
- 上のレイヤー（L4・L5）が常に「良い」わけではない
- 課題・コスト・データ量・技術リソースで最適なレイヤーが決まる
- まずL1から始め、課題が明確になってからL2〜L5へ段階的に進む

### S14 レイヤー1：プロンプト設計の3原則
**ヘッドライン:** 意図通りの回答を得るには「前提・役割・出力形式の明確化」と「段階的な指示」を設計する

**原則①｜前提・依頼・出力形式の明確化**
- **前提（コンテキスト）:** 対象・範囲・禁止事項を明示
  - 例: 「これは法人向けの研修教材です。専門用語は避けてください。」
- **依頼（インストラクション）:** 具体的なアクション指示
  - 例: 「以下の会議録を3点に要約してください」
- **出力形式:** 箇条書き・表・文字数・フォーマットを指定
  - 例: 「各項目50文字以内・箇条書き3点で」

**原則②｜役割付与と段階的指示で制御**
- 役割付与: 「○○向けに」でトーン・視点を調整
  - 例: 「営業担当者向けに、技術的な説明を避けて」
- 段階的指示: 複雑な課題を分割して依頼
  - 例: Step1で分析 → Step2で整理 → Step3でまとめ

**原則③｜曖昧さと詰め込みすぎを避ける**
- 曖昧な指示: 「良い文章にして」→ 的外れな出力の原因
- 詰め込みすぎ: 一度に10以上の条件 → 守られない条件が増える
- → 1プロンプト1タスクを基本とする

- **図解パターン:** module-block

### S16 レイヤー2：汎用AIの「API利用」─業務フローへの組み込み
**ヘッドライン:** API利用の本質は「プログラムから自動でAIを呼び出す」ことで、手作業では不可能な大量処理や業務システムへのシームレスな組み込みを実現する

**APIとは何か:**
- API（Application Programming Interface）= 異なるソフトウェアが「プログラム経由」で機能を連携させるための「窓口・約束ごと」
- 「ツール利用」が人間の操作 → 「API利用」はプログラムが自動で呼び出す

**ツール利用とAPI利用の本質的な違い:**

| 観点 | ツール利用 | API利用 |
|---|---|---|
| 呼び出し主体 | 人間がブラウザ・アプリを開いて操作 | プログラムが自動で呼び出す |
| 処理の単位 | 1回ごとに人が確認・コピー | バッチ処理・自動連携・大量処理が可能 |
| 費用体系 | 月額固定（ChatGPT Plus等） | 従量課金（トークン数に応じて課金） |
| カスタマイズ | プロンプトの工夫のみ | システムに応じた細かい設計・パラメータ指定が可能 |

**API利用が必要になるタイミング:**
- 1万件以上のデータを一括処理したい
- 自社システムにAI機能を組み込みたい
- 毎晩自動でレポートを生成させたい

- **図解パターン:** comparison-contrast

### S17 レイヤー2：APIの仕組みと呼び出し方
**ヘッドライン:** APIはソフトウェア同士が「プログラム経由で機能を共有する仕組み」。リクエストを送るとレスポンスが返ってくる

**APIの3層構造:**

① **ソフトウェア間の「窓口」**
- あるソフトウェアの機能を、別のプログラムから呼び出すための仕様
- 例: 天気予報アプリが「気象データ提供API」を呼び出して最新天気を取得

② **「ツール」と「API」の違い**
- ツール/アプリ: 人間が画面操作（GUI）→ 結果を受け取る
- API: プログラムが仕様に従って呼び出す → 結果を受け取る

③ **APIの呼び出しとパラメータ**
- コール（呼び出し）: 具体的な依頼（パラメータ）を送信
  - 例: 「この文章を要約して、100文字で」
- レスポンス: 結果データ（要約文）が返ってくる

**身近なAPI活用例:**
- 天気予報サイト → 気象庁APIを呼び出して気象データを取得・表示
- ECサイト → 決済サービスAPIを呼び出して支払い処理
- 自社アプリ → 地図サービスAPIを呼び出して地図を表示

- **図解パターン:** process-flow

### S18 レイヤー2：API利用の主なビジネスシナリオ
**ヘッドライン:** APIを活用することで、自社システムへのAI機能の組み込みと、手作業では不可能な大量データの一括処理（バッチ処理）が実現できる

**シナリオ①｜自社サービス・システムへの組み込み**

流れ: 自社Webサイト/業務システム → APIリクエスト → 生成AI（リアルタイム処理） → 結果反映

### S19 レイヤー3：RAG（検索拡張生成）─自社ナレッジを精通したAI
**ヘッドライン:** RAGは「自社文書・PDFを検索可能な形で保持し、質問時に関連情報を取得してから回答を生成する」方式で、自社ナレッジ連携の現在最適かつ主流の戦略的解法

**RAGが必要な理由:**
- 汎用LLMは学習データ（公開情報）しか知らない
- 社内マニュアル・規定・顧客情報・過去事例を「知っているAI」を作るには自社データとの連携が必要
- Fine-tuningより低コスト・データ更新が容易

**RAGの仕組み（6ステップ）:**

```
0. 社内ドキュメントをデータベース化（ベクトル化・インデックス）
1. ユーザーが質問を入力
2. 質問を検索クエリに変換
3. ナレッジデータベースを検索
4. 関連情報を取得
5. プロンプト（質問+関連情報）をLLMに送信
6. LLMが関連情報を根拠に回答を生成
```

**ノートブック型・ストレージ連携型・RAGの比較:**

| 方式 | 規模 | 精度 | 運用 |
|---|---|---|---|
| ノートブック型 | 少量・その場 | 限定的 | 手動 |
| ストレージ連携型 | 中規模 | 中程度 | 自動連携 |
| RAG構築 | 全ナレッジ対応 | 高精度・自社最適 | 設計・運用が必要 |

- **図解パターン:** process-flow

### S23 レイヤー4：AIエージェント─単なる対話AIを超えた自律実行
**ヘッドライン:** AIエージェントは「質問に答える」だけでなく、目標達成に向けてタスクを自ら分解・計画し、ツールを呼び出して実行する「作業者型AI」

**エージェントの4要素:**

① **計画（Planning）**
- 与えられた目標を達成するためのステップを自律的に計画
- 例: 「競合レポートを作成して」→ 検索→収集→分析→執筆の計画を自動立案

② **ツール実行（Tool Use）**
- 必要なツール（検索・DB・メール・RPA・外部API）を選択・呼び出し
- 複数ツールを連携させた複合タスクの実行

③ **結果確認（Observation）**
- ツール実行の結果を評価し、計画通りか確認
- 予期しない結果の場合は計画を修正

④ **ネクストアクション（Next Action）**
- 確認結果に基づいて次のアクションを判断
- 必要に応じてユーザーへの確認を求める

**連携できるツールの例:**
- データ分析ツール / メールクライアント / ワークフロー承認システム / CRMシステム

- **図解パターン:** continuous-operation-loop

### S24 レイヤー4：AIエージェントによる業務プロセス自動化の4パターン
**ヘッドライン:** 予算申請・問い合わせ対応・月末レポート・社内振り分けなど、複数の手順とシステム連携を伴う定型業務をエージェントが自動遂行できる

**パターン①｜予算申請のたたき台自動作成**
- トリガー: 申請開始

**パターン②｜問い合わせ対応の半自動化**
- トリガー: 問い合わせ受信

**ツール利用との違い:** エージェントは「組織のワークフローに統合され、システムイベントをトリガーに自動実行」される点が根本的な差異

### S25 レイヤー5：ファインチューニング─自社データで再学習させる独自AI
**ヘッドライン:** ファインチューニングは「元のLLMに自社データを追加学習」させて、特定ドメイン・業務に特化したカスタムモデルを構築する最上位のアプローチ

**ファインチューニングとRAGの違い:**

| 観点 | RAG | ファインチューニング |
|---|---|---|
| 仕組み | 検索で情報を取得して回答 | モデル自体を自社データで再学習 |
| データ更新 | ドキュメントを追加するだけ | 再学習が必要（コスト・時間） |
| 向いている用途 | 最新情報・社内文書参照 | 固定文体・専門用語・形式の統一 |
| コスト | 中 | 高（GPUリソース・専門知識） |
| 独自性 | 中 | 高（モデル自体が自社仕様に変わる） |

**ファインチューニングのプロセス:**
1. 元のモデル（汎用LLM: GPT等）を準備
2. 自社データ（追加学習用データ）を準備: 社内文脈・独自表現・専門用語
3. ファインチューニング（追加学習）: モデルパラメータを再設計・最適化
4. 自社向けカスタムモデル: 企業の文脈・表現に適合・高精度ドメイン知識

- **図解パターン:** process-flow

### S26 レイヤー5：ファインチューニングが有効な4つのユースケース
**ヘッドライン:** ファインチューニングは独自性を高めるが難易度・コストが高いため、専門用語・固定文体・更新頻度の低い領域など効果的なユースケースに限定すべき

**ユースケース①｜固定された文体・フォーマットが必要な文章生成**
- 用途: 報告書・公式文書・契約書のたたき台
- 理由: 社内で統一されたトーン・表現にしたい場合 / RAGでは文体の統一が難しい

**ユースケース②｜専門領域での高精度な応答**
- 用途: 法務・医療・金融など業界特有の用語・論理が厳密に必要な場合
- 理由: 汎用LLMでは専門用語の解釈精度が不十分な場合がある

**ユースケース③｜企業独自の言い回し・スタイルの確立**
- 用途: ブランドのトーン・顧客向けの応答スタイル
- 理由: 「この会社らしい言い方」をモデルに持たせたい場合

**ユースケース④｜業務ルールが固定で更新頻度が低い領域**
- 用途: 同じパターンの入出力を大量に扱う処理
- 理由: 学習コストをかける価値がある反復性の高い業務

**注意:** 更新頻度が高い情報にはRAGの方が適している

- **図解パターン:** module-block

### S30 AI・生成AI・LLMの3層関係
**ヘッドライン:** AI（人工知能）という広範な領域の中に生成AIが位置づけられ、さらにその中核技術としてLLM（大規模言語モデル）が存在する

**3層の包括関係（詳細版）:**

**AI（人工知能）──最も広い概念**

- 現在の主流: 機械学習・深層学習

**LLM（大規模言語モデル）──生成AIの中核技術**
- 膨大なテキストデータを学習した深層学習モデル
- テキスト生成に特化

- 基盤技術: Transformer アーキテクチャ（2017年登場）
- 代表: GPT-4o・Gemini 2.0・Claude Sonnet・LLaMA 4

- **図解パターン:** hierarchy-pyramid

### S31 AIの技術的進化史：統計→機械学習→ディープラーニング→生成AI
**ヘッドライン:** 生成AIの源流は統計解析・機械学習にある。技術革新の積み重ねが現在の高度な生成能力へと進化した

### S36 ツール利用との違い：「使う」から「組み込む」への転換
**ヘッドライン:** ツール利用は「個人のタスクをその場で処理」するが、業務実装（エージェント）は「組織のワークフローに統合され、自動実行」される点が根本的な差異

**2つの活用形態の比較:**

**【ツール利用（使う）】個人のタスクをその場で依頼**
- 起点: 人がツールを開いて操作（セッション依存）
- 範囲: ツールが提供する機能内（検索・ファイル操作等）で完結
- 連携: 基本的にツール内に閉じた動作
- 目的: 個人の生産性向上・スポット的なタスク処理

**【業務実装（組み込む）】組織のワークフローに統合**
- 起点: システムイベント・スケジュール・他トリガー（人手不要）
- 範囲: 自社の業務フロー全体・複数システムに跨る処理
- 連携: 社内基幹システム・メール・承認ツール等と接続
- 目的: 組織的な自動化・定型・反復業務の削減・ビジネス価値創出

**体感・検証フェーズ → 本格導入・展開フェーズへの移行がポイント**

- **図解パターン:** comparison-contrast

### S37 API活用したバッチ処理例①：VoC・アンケートの自動タグ付け

4. 結果の反映・可視化（自社画面・ダッシュボードにシームレスに反映・インサイト抽出）

**効果（想定例）:**
- 処理時間: 人手5日 → 自動3時間（夜間バッチ）
- 分析カバレッジ: 20%抽出 → 100%全件

### S41 クラウドAI vs ローカルAI：選択基準と使い分け
**ヘッドライン:** クラウドAI（API）とローカルAI（LLM）は目的・環境・コストで使い分ける。多くの場合はクラウドAIで十分

**比較表:**

| 比較項目 | クラウドAI（API） | ローカルAI（LLM） |
|---|---|---|
| 実行環境 | ベンダー提供のサーバー | 自社サーバー / PC |
| データの流れ | インターネット経由で外部送信 | 社内ネットワーク内で完結 |
| オフライン利用 | 不可 | 可能 |
| 最新モデル | 常に最新（ベンダーが更新） | 自分でモデル選択・更新 |
| 費用 | 従量課金（トークン数） | 初期サーバーコスト・管理コスト |
| セキュリティ | ベンダーのセキュリティポリシーに依存 | 自社管理（責任も自社） |

**ローカルAIが選択肢になる3つの条件:**

① **セキュリティ・コンプライアンス:** 機密データを外部クラウドに送れない規制環境（金融・医療・官公庁等）

② **コスト最適化:** 大量・継続的なAPI処理でコストが膨らむ場合（社内文書の再インデックス・感情分析等）

③ **オフライン環境:** 工場・現場・オフショアなどネット接続を前提にしたくない場所

- **基本方針:** まずクラウドAIで始め、上記3条件に当てはまる場合にローカルAIを検討

- **図解パターン:** comparison-contrast

### S44 データサイエンスと生成AIの役割分担
**ヘッドライン:** 数値予測・最適化が得意な「データサイエンス」と、言語・非構造化データの処理が得意な「生成AI」は、相互補完の関係にある

**2つの強みの比較:**

| 観点 | データサイエンス | 生成AI |
|---|---|---|
| 得意な入力 | 数値・表形式の構造化データ | テキスト・画像・音声の非構造化データ |
| 主な処理 | 統計・機械学習・数理最適化 | 言語理解・生成・対話・オーケストレーション |
| 典型的なアウトプット | 数値・スコア・最適解・グラフ | 文章・要約・対話・実行結果 |
| 典型的なユースケース | 需要予測・離脱予測・レコメンド・在庫最適化 | 議事録要約・問い合わせ対応・レポート生成 |

**「代替」ではなく「役割分担と連携」が正しい視点:**
- 生成AIがデータサイエンスを置き換えるのではない
- 役割分担と連携で業務の価値を高めるという視点が大切
- 両者が補完しあう領域（③）を意識的に設計することが高度なAI活用へつながる

### S47 Text-to-SQL：自然言語でデータベースを操作する
**ヘッドライン:** Text-to-SQLは「自然言語の質問をSQLに自動変換し、データベース実行まで行う」技術で、エンジニアを介さずに現場担当者がデータを自分で抽出できる「データ活用の民主化」を実現する

**Text-to-SQLの5ステップ:**
1. ユーザーが自然言語で質問（「今月の問い合わせ数を部署別に見たい」）
2. LLMがDBスキーマを参照（テーブル名・カラム名・関係性を「コンテキスト」として受け取る）
3. LLMがSQL文を生成（`SELECT department, COUNT(*) FROM inquiries WHERE month='2025-03' GROUP BY department`）
4. データベース上で実行
5. 結果が返される（「営業部が最多で150件です」）

**データ活用の民主化の意義:**
- エンジニアやアナリストを介さず、現場担当者が自らデータを抽出・活用できる
- 意思決定のスピードが上がる

**Text-to-SQLが組み込まれた主要BIツール（2025年）:**
- Tableau Pulse / Metabase / Snowflake Cortex Analyst / Databricks Genie / ThoughtSpot

### S48 マルチモーダルAIによる非構造化データの構造化

### S49 補強：APIの定義を「窓口・約束ごと・呼び出し」で理解する
**ヘッドライン:** APIはアプリ同士が機能を呼び出すための窓口であり、生成AIを業務システムに組み込む入口になる

| 観点 | 説明 | 生成AI APIでの例 |
|---|---|---|
| 窓口 | 外部プログラムから機能を呼び出す場所 | チャット生成、要約、分類 |
| 約束ごと | 入力形式、認証、返却形式のルール | APIキー、JSON、モデル指定 |
| 呼び出し | プログラムが依頼を送る動作 | 「この文章を分類して」と送信 |
| 返却 | AIの処理結果を受け取る | 分類結果、理由、信頼度 |

- **業務での意味:** 人がチャット画面を開かなくても、システムが必要なタイミングでAIを呼び出せる
- **図解パターン:** process-flow

### S50 補強：APIレスポンスは業務データへ書き戻して初めて価値になる
**ヘッドライン:** API利用では「AIに聞く」だけでなく、レスポンスを表・DB・通知・画面へ戻す設計が重要になる

**API処理の流れ:**
1. 業務データを取得する（CSV、DB、フォーム入力）
2. AIに渡す入力を整形する（プロンプト、制約、出力形式）
3. APIへリクエストを送る
4. レスポンスを受け取る（文章、JSON、分類ラベル）
5. 結果を業務システムへ書き戻す
6. 人が確認し、必要に応じて修正する

| 失敗しやすい点 | 対策 |
|---|---|
| 出力形式が毎回違う | JSON形式や選択肢を指定する |
| エラー時に処理が止まる | リトライ、ログ、未処理一覧を設計する |
| AI結果を誰も確認しない | 承認ステータスを追加する |

- **図解パターン:** process-flow

### S51 補強：APIの身近な利用例から生成AI活用を逆算する
**ヘッドライン:** 天気、決済、地図のAPIと同じように、生成AI APIも既存業務の裏側で自動処理を担う部品として考える

| 身近なAPI | 何をしているか | 生成AI APIへの置き換え |
|---|---|---|
| 天気API | 地域・日時を渡し、天気を返す | 文書を渡し、要約を返す |
| 決済API | 金額・カード情報を渡し、決済結果を返す | 問い合わせを渡し、分類結果を返す |
| 地図API | 住所を渡し、地図・経路を返す | 商品情報を渡し、説明文を返す |
| 翻訳API | 原文を渡し、翻訳文を返す | 議事録を渡し、ToDoを返す |

- **設計の問い:** 自社業務で「入力→処理→出力」が決まっている繰り返し作業はどこか
- **図解パターン:** comparison-contrast

### S52 補強：ツール利用とAPI利用は費用・責任・運用が異なる
**ヘッドライン:** API利用は便利だが、従量課金、ログ、エラー処理、権限管理まで含めて運用設計しなければ業務に組み込めない

| 観点 | ツール利用 | API利用 |
|---|---|---|
| 利用者 | 個人・チーム | システム・業務フロー |
| 費用 | 月額プラン中心 | 従量課金、上限管理が必要 |
| 品質管理 | 人が画面で確認 | 自動処理後にレビュー工程を設計 |
| エラー対応 | その場で再実行 | ログ、リトライ、通知が必要 |
| セキュリティ | 利用規約・入力ルール | 認証情報、アクセス制御、監査ログ |

- **判断基準:** 同じ処理を大量・定期・システム内で行うならAPI利用を検討する
- **図解パターン:** comparison-contrast

### S53 補強：RAGの仕組みは「検索してから答える」2段階で理解する
**ヘッドライン:** RAGはAIに社内文書を丸暗記させるのではなく、質問に関連する情報を検索し、その根拠を使って回答を生成する方式

**RAGの基本ステップ:**
1. 文書を分割し、検索できる形にする
2. ユーザーの質問を受け取る
3. 質問に近い文書片を検索する
4. 検索結果をLLMに渡す
5. LLMが根拠に基づいて回答する
6. 出典・参照箇所を返す

| RAGで改善できること | それでも残る課題 |
|---|---|
| 未公開の社内情報を参照できる | 参照元文書が古いと回答も古い |
| 根拠付き回答にしやすい | 検索に失敗すると回答品質が落ちる |
| Fine-tuningより更新しやすい | 権限・更新・評価の運用が必要 |

- **図解パターン:** process-flow

---

### S54 補強：ナレッジ基盤はRAG以前に「何を正とするか」を決める仕組み
**ヘッドライン:** RAGの精度はAIモデルだけで決まらず、正しい資料、更新ルール、権限、評価データを整えるナレッジ基盤で決まる

| 設計項目 | 決めること | 不備があると起きること |
|---|---|---|
| 正本資料 | どの文書を公式情報とするか | 古い手順を回答する |
| 更新ルール | 誰がいつ差し替えるか | 部署ごとに回答がずれる |
| 権限 | 誰がどの資料を参照できるか | 機密情報が漏れる |
| 評価 | 正答例・NG例をどう用意するか | 改善できない |

- **業務への示唆:** RAG導入前に「資料棚卸し」と「参照権限の整理」を行う
- **図解パターン:** structure-layer

### S55 補強：生成AI基盤はデータ収集から実行層までを一連の流れで設計する
**ヘッドライン:** 生成AI基盤はチャット画面ではなく、データソース、検索、LLM、ツール連携、評価をつなぐ業務インフラとして設計する

**基盤のレイヤー:**
- **データソース:** 社内文書、FAQ、議事録、業務DB、Web公開情報
- **前処理:** 分割、メタデータ付与、重複除去、権限ラベル付け
- **検索基盤:** ベクトル検索、キーワード検索、再ランキング
- **生成層:** LLM、プロンプト、回答形式、引用表示
- **実行層:** API、エージェント、通知、ワークフロー承認
- **評価・運用:** ログ、正答率、改善サイクル、権限監査

- **図解パターン:** structure-layer

### S56 補強：RAG高度化は「検索品質」と「知識品質」の両方を改善する
**ヘッドライン:** RAGは導入したら終わりではなく、検索で見つける力と、参照する知識そのものの質を継続的に高める必要がある

| 改善対象 | 初期状態 | 高度化の方向 |
|---|---|---|
| 検索品質 | 単純な類似検索 | ハイブリッド検索、再ランキング、質問変換 |
| 知識品質 | PDFをそのまま投入 | 構造化、タグ付け、正本管理、古い資料削除 |
| 回答品質 | 文章を返すだけ | 出典表示、確信度、未確認事項の明示 |
| 運用品質 | 利用ログなし | 評価データ、失敗事例、改善担当者 |

- **到達点:** 全社で使えるRAGは、AI開発ではなくナレッジ運用の仕組みとして育てる
- **図解パターン:** continuous-operation-loop

### S57 補強：ファインチューニングの仕組みは「知識追加」より「振る舞い調整」に近い
**ヘッドライン:** Fine-tuningはモデルに最新資料を検索させる手段ではなく、回答形式、専門用語、分類基準などの振る舞いを安定させる手段として考える

| 目的 | RAGが向く | Fine-tuningが向く |
|---|---|---|
| 最新資料を参照したい | ○ | △ |
| 社内文書の根拠を示したい | ○ | △ |
| 回答形式を安定させたい | △ | ○ |
| 分類基準を学習させたい | △ | ○ |
| 独自の文体・専門用語を揃えたい | △ | ○ |

**注意点:**
- 学習データの品質が低いと、モデルの振る舞いも悪化する
- 更新頻度が高い知識はRAGで扱う方が運用しやすい
- コスト、評価、再学習の計画が必要

- **図解パターン:** comparison-contrast

### S58 補強：Text-to-SQLは現場担当者のデータ抽出を支援する

**処理の流れ:**

### S62 補強：生成AIとデータサイエンスの組み合わせは「言語」と「数値」を分担する
**ヘッドライン:** 生成AIは言語・文書・非構造化データに強く、データサイエンスは数値・予測・最適化に強い。両者をつなぐと業務価値が高まる

| 役割 | 生成AIが得意 | データサイエンスが得意 |
|---|---|---|
| 入力 | 文書、画像、音声、問い合わせ | 売上、在庫、顧客属性、時系列 |

| 出力 | 説明文、提案書、FAQ回答 | スコア、予測値、セグメント、最適案 |
| 連携例 | 問い合わせ文を分類 | 分類結果から傾向分析 |

- **実務の型:** 生成AIで非構造化データを構造化し、データサイエンスで分析し、再び生成AIで説明・提案に変換する
- **図解パターン:** network-relationship

### S63 第2回まとめ：4つの要点と今後の展望
**ヘッドライン:** 生成AI活用の本質は「汎用ツールを使う」から「自社ナレッジと組み合わせて競争優位を作る」への転換にある

**4つの要点:**

**① 汎用AIツール利用は通過点（第1章）**
- 「ChatGPTを使っている」では差別化にならない
- 自社固有のナレッジ・データ × 生成AI = 競争優位性

**② 5レイヤーの理解と選択（第2章）**
- L1〜L5の特徴・コスト・難易度を理解した上で、自社の課題・資源に合ったレイヤーを選択
- まずL1から始め、段階的に高度化する

**③ LLMの仕組みとプロンプト設計（第3章）**
- LLMは「確率的にもっともらしい文を生成」
- プロンプトの3原則（前提・役割・形式）でアウトプット品質が変わる

**④ 「使う」から「組み込む」への転換（第4章）**
- バッチ処理・システム組み込みで「組織のワークフロー」にAIを統合
- ローカルLLM・データサイエンスとの融合で高度化

- **図解パターン:** module-block
